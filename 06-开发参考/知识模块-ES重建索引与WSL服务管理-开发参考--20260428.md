## 一、ES 从 MySQL 重建索引问题

### 1.1 问题描述

`reindex_es.py` 脚本从 MySQL 重建 ES 索引时，所有 38 条知识全部失败，错误：

```
{"error":{"root_cause":[{"type":"document_parsing_exception","reason":"[1:1190] failed to parse field...
```

### 1.2 已尝试的修复

1. **控制字符清理**：`clean_for_json()` 函数去除 `\x00-\x08\x0b\x0c\x0e-\x1f` 等控制字符
2. **MySQL charset**：使用 `utf8mb4` 编码连接
3. **JSON 序列化**：使用 `ensure_ascii=False` 避免转义

**结果**：仍全部失败

### 1.3 进一步排查方向

错误位置 `[1:1190]` 表示 JSON 第 1 行第 1190 个字符附近。问题可能是：

1. **JSON 中存在未转义的特殊字符**（反斜杠、引号、换行符）
2. **MySQL 内容包含破坏 JSON 结构的字符**
3. **curl 命令的 JSON 传递方式问题**

### 1.4 调试方法

```python
# 在 reindex_es.py 中添加调试
json_str = json.dumps(doc, ensure_ascii=False)
# 打印到文件检查
with open('debug_doc.json', 'w', encoding='utf-8') as f:
    f.write(json_str)

# 然后用 curl 直接测试
curl -X PUT "http://172.20.36.91:9200/knowledge/_doc/test?refresh=true" \
  -H "Content-Type: application/json" \
  -d @debug_doc.json
```

### 1.5 可能的解决方案

1. **使用 Python ES 客户端** 代替 curl 命令，避免 JSON 序列化问题
2. **对 content 字段进行更严格的过滤**：移除所有非打印字符
3. **分段测试**：逐条测试找出哪条有问题

### 1.6 相关文件

| 文件 | 说明 |
|------|------|
| `reindex_es.py` | 重建索引脚本（有问题） |

## 二、WSL systemd 服务配置

### 2.1 问题背景

每次启动 WSL 后，Docker 服务需要手动启动。为实现自动化，配置 systemd user 服务。

### 2.2 服务文件

**路径**：`06-开发参考/docker-services.service`

```ini
[Unit]
Description=Docker Services for Knowledge Base

[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/usr/bin/docker start redis qdrant elasticsearch neo4j
ExecStop=/usr/bin/docker stop redis qdrant elasticsearch neo4j

[Install]
WantedBy=default.target
```

### 2.3 安装步骤

```bash
# 复制到 systemd 用户目录
cp docker-services.service ~/.config/systemd/user/

# 启用服务（自动启动）
systemctl --user enable docker-services.service

# 启动服务
systemctl --user start docker-services.service

# 检查状态
systemctl --user status docker-services.service
```

### 2.4 验证

```bash
# 检查容器状态
docker ps

# 应该看到 4 个容器运行：
# redis, qdrant, elasticsearch, neo4j
```

## 三、Neo4j 数据清理

### 3.1 问题

Neo4j 中存在没有 `EXTRACTED_FROM` 关系的孤立节点（标签为 None 的 `146` 个节点）。

### 3.2 清理脚本

**文件**：`fix_neo4j.py`

```python
from neo4j import GraphDatabase

driver = GraphDatabase.driver('bolt://172.20.36.91:7687')
with driver.session() as session:
    # 删除所有没有 EXTRACTED_FROM 关系的节点
    result = session.run('''
        MATCH (n)
        WHERE NOT EXISTS(()-[:EXTRACTED_FROM]->(n))
        DETACH DELETE n
        RETURN count(n) as deleted
    ''')
    count = result.single()['deleted']
    print(f'Deleted: {count} nodes')
```

### 3.3 检查脚本

**文件**：`check_neo4j.py`

## 五、"第78条"幻觉问题

### 5.1 问题现象

无论问什么问题，推理链的"检索相关知识库内容"步骤里都会出现"第78条"。

### 5.2 调查结论

**确认为 LLM 幻觉，非数据问题**：
- MySQL 知识内容中无"第78条"相关内容
- ES 检索结果中也没有"第78条"
- 这是 LLM 在生成推理过程描述时自行编造的

### 5.3 可能的解决方案

1. **改进 Prompt**：在 Prompt 中强调"基于真实检索结果，不要编造"
2. **后端校验**：在返回推理链前检查是否与实际检索结果匹配
3. **直接使用检索结果**：不在推理链中让 LLM 自由发挥
