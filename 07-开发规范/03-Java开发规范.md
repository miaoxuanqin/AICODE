# Java开发规范

## 1. 文档目的

用于统一 Java 服务端工程的架构设计、代码组织、命名、分层、异常处理和测试要求。

## 2. 适用范围

- Java 后端服务
- 平台服务
- 集成服务
- 数据服务
- 调度与任务执行服务

## 3. 总体原则

- 结构清晰
- 分层明确
- 接口稳定
- 易于维护
- 易于测试
- 易于扩展

## 4. 工程结构规范

### 4.1 模块结构

- 按业务边界拆分模块，禁止以技术框架名随意拆模块
- 优先采用多模块结构：`app`、`api`、`domain`、`infrastructure`、`common`
- `app` 负责启动、装配、配置、对外暴露接口
- `api` 负责 DTO、Feign 接口、事件定义等跨模块契约
- `domain` 负责核心领域对象、领域服务、业务规则
- `infrastructure` 负责数据库、缓存、消息、三方服务等技术实现
- `common` 仅承载稳定且通用的工具、常量、基础异常，不承载具体业务
- 模块之间依赖方向必须清晰：上层依赖下层，业务模块不反向依赖启动模块
- 公共模块不得演变为“大杂烩”工具箱，不相关能力不得强行沉淀到 `common`

### 4.2 包结构

- 包名统一使用小写英文单词，按领域或职责分层组织
- 推荐包结构：`controller`、`service`、`domain`、`repository`、`mapper`、`model`、`dto`、`config`、`exception`
- 包结构优先体现业务语义，其次再体现技术分层
- 同一包内类职责应高度相关，避免一个包同时混入接口、实现、DTO、配置
- `impl` 包仅用于接口实现，不得放 DTO、实体、工具类
- `util` 包仅允许放明确无状态、无业务语义的辅助类；复杂逻辑应进入领域或服务层

### 4.3 分层规范

- Controller 只负责协议转换、参数接收、调用应用服务、返回统一响应
- Controller 不编写核心业务逻辑，不直接操作数据库，不拼接复杂查询条件
- Controller 必须进行入参校验、权限校验接入和错误响应映射
- Service 负责用例编排、事务边界控制、跨领域协调
- Service 方法应围绕一个清晰业务动作设计，避免“大而全”的万能服务
- Service 层可以组合多个 Repository 或 Domain Service，但不承载底层 SQL 细节
- Domain 负责核心业务规则、状态变更约束、领域对象行为
- 领域对象应尽量表达业务语义，而不是仅作为贫血数据结构传递
- 多处复用且与业务强相关的规则，应下沉到 Domain 层统一维护
- Repository / DAO 仅负责数据持久化与查询映射
- Repository / DAO 返回结果应尽量靠近领域语义，避免将底层数据库字段含义泄漏到上层
- Repository / DAO 不混入业务判断、权限判断、流程控制

## 5. 编码规范

### 5.1 命名规范

- 类名使用 UpperCamelCase，方法、变量、参数使用 lowerCamelCase
- 常量使用全大写加下划线，如 `MAX_RETRY_COUNT`
- 包名全部小写，禁止中英文混用、拼音缩写、无语义缩写
- 接口命名使用业务名词或能力名词，如 `OrderService`
- 实现类以 `Impl` 结尾，如 `OrderServiceImpl`
- 抽象类以 `Abstract` 或 `Base` 开头
- 异常类以 `Exception` 结尾，枚举类以 `Enum` 结尾
- 方法命名应体现动作语义：查询单个用 `get` / `find`，列表查询用 `list`，统计用 `count`，新增用 `create` / `save`，删除用 `remove` / `delete`
- 布尔字段避免使用 `isXxx` 作为属性名，统一用 `enabled`、`deleted`、`success` 等可读表达

### 5.2 对象设计

- DTO、VO、DO、Entity、Command、Query 的职责必须区分，不得混用
- 持久化对象不直接暴露给接口层，接口请求和响应必须使用独立 DTO
- 对象字段应保持最小必要集合，避免为了省事传递“大对象”
- POJO 成员优先使用包装类型，便于表达空值和未赋值状态
- 禁止在对象属性上设置会掩盖真实状态的默认值
- 构造函数中不编写复杂业务逻辑；初始化逻辑应放在工厂方法、Builder 或显式初始化方法中
- 对外暴露的对象应实现稳定的 `toString`，但不得输出密码、令牌、密钥等敏感字段
- 值对象优先保持不可变；需要修改状态的对象应通过明确方法完成变更

### 5.3 接口设计

- 接口优先面向业务能力设计，不暴露底层实现细节
- 公共接口一旦发布应保持兼容，非必要不得修改字段语义和返回结构
- 过期接口必须使用 `@Deprecated` 标识，并说明替代方案和下线计划
- 方法参数不超过 3 个；超过时应封装为命令对象或查询对象
- 返回值语义必须明确，不使用 `Map<String, Object>` 作为通用返回结构
- 接口定义必须补充注释，说明用途、参数含义、返回约束和异常情况
- 同一接口中不得混合查询和写操作语义
- 空集合返回空列表，不返回 `null`

### 5.4 异常处理

- 异常分为业务异常和系统异常，禁止全部混用 `RuntimeException`
- 可预期的业务失败使用明确的业务异常类型和稳定错误码
- 不得吞异常；捕获异常后必须记录上下文并决定转换、补偿或继续抛出
- 不得使用异常做普通流程分支控制
- 抛出的异常信息应可定位问题，但不得泄露密码、密钥、SQL 明文等敏感信息
- Controller 层统一通过全局异常处理器转换为标准响应
- 外部依赖调用失败时，应补充依赖标识、请求关键参数和超时信息

### 5.5 日志规范

- 日志级别统一使用 `ERROR`、`WARN`、`INFO`、`DEBUG`，禁止滥用 `ERROR`
- `INFO` 用于关键流程节点和状态变化，`DEBUG` 用于排查细节，生产默认可关闭
- 记录日志必须包含业务标识，如 `requestId`、`traceId`、`orderId`、`taskId`
- 使用参数化日志，禁止字符串拼接
- 敏感信息默认脱敏，不打印密码、Token、身份证、手机号全量、密钥原文
- 高频路径、循环内部、批量处理场景应控制日志量，避免刷屏
- 异常日志必须保留异常栈，不得只打印 `e.getMessage()`

## 6. API 与 DTO 规范

- 请求对象：必须独立定义 DTO，字段名清晰，使用校验注解约束必填、长度、格式和范围；查询条件与写入命令分开建模
- 响应对象：统一返回结构，至少包含成功标记、业务码、消息、数据体；响应字段稳定后不得随意重命名
- 分页对象：统一分页请求与响应模型，明确 `pageNo`、`pageSize`、`total`、`items`；对 `pageSize` 设置上限，防止大查询
- 错误码规范：错误码应按模块分段管理，可读、稳定、可检索；业务错误码与 HTTP 状态码分层表达，不得仅依赖错误文案识别失败原因
- DTO 不直接暴露内部枚举编码细节；必要时同时返回 `code` 和 `label`
- 接口时间字段统一使用 ISO 8601 或明确约定的标准格式，禁止多种格式混用
- 金额、比例、精度敏感数值优先使用 `BigDecimal`

## 7. 数据访问规范

- 主键与 ID 生成：所有业务表主键统一使用雪花 ID，不使用数据库自增 ID
- ORM / SQL 使用原则：简单单表操作优先使用 ORM；复杂查询可使用显式 SQL，但必须可读、可维护、可审计；禁止在代码中拼接未校验 SQL 片段
- 事务管理：事务边界定义在 Service 层；只为必要写操作开启事务；禁止将远程调用、长耗时计算包裹在大事务中
- 批量处理：批量写入、更新、删除必须分批执行，避免单次事务过大；批量任务需要幂等设计和失败重试策略
- 性能要求：查询必须命中合理索引，禁止无条件全表扫描进入核心链路；列表查询必须限制返回数量；大字段按需读取，避免 `select *`
- Repository 方法命名和 SQL 语义保持一致，避免“方法名查单条、SQL 实际查多条”
- 更新操作必须明确条件，禁止无条件更新或删除
- 涉及状态流转的数据更新应考虑并发控制，可使用版本号或条件更新
- 数据库时间、字符集、排序规则、空值语义在同一服务内保持一致

## 8. 安全规范

- 鉴权接入：所有对外接口默认需要鉴权，白名单接口必须显式声明并经过评审
- 数据权限：除功能权限外，还必须校验数据归属、租户边界、组织边界和资源可见范围
- 参数校验：所有外部输入默认不可信，必须进行格式、长度、枚举值、边界和非法字符校验
- 敏感操作审计：登录、授权、导出、删除、配置变更、批量修改等动作必须记录审计日志
- 配置中的密码、密钥、令牌必须通过密文或密钥管理系统注入，禁止硬编码
- 文件上传、URL 回调、脚本执行、表达式解析等高风险入口必须做额外安全控制
- 对外错误信息避免暴露内部类名、表名、SQL 结构和中间件拓扑

## 9. 测试规范

- 单元测试：核心业务规则、边界条件、异常分支必须覆盖；单元测试应快速、稳定、可重复执行
- 集成测试：覆盖数据库、缓存、消息、配置装配、事务行为和外部依赖适配层
- 接口测试：覆盖成功、失败、鉴权失败、参数非法、分页边界、幂等等关键场景
- 性能测试：对高并发、高频查询、批量任务、报表类接口进行容量评估和压测验证
- 新增功能默认同步补充测试；修复缺陷必须补充回归用例
- 测试命名应清晰表达场景和预期结果，避免无语义缩写
- 测试数据应最小化且可控，不依赖共享环境中的脏数据
