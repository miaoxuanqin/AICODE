> 创建日期：2026-04-27
> 更新日期：2026-04-28

---

## 1. 概述

图谱增强 RAG（Graph-RAG）在传统 RAG 基础上引入知识图谱，通过实体和关系建立知识之间的关联，在问答时展示多跳推理路径，增强回答的可解释性和完整性。

**与向量 RAG 的区别**：

| 维度 | 向量 RAG | 图谱增强 RAG |
|------|----------|--------------|
| 检索方式 | 语义相似度 | 图遍历 + 语义 |
| 推理能力 | 单跳 | 多跳 |
| 结果展示 | 知识卡片列表 | 推理路径图 |
| 可解释性 | 低 | 高 |

**Neo4j 集成方案（2026-04-28 新增）**：

采用**向量检索 + Neo4j 图数据库结合**的混合架构：
- 向量检索负责"广撒网"快速找到相关文档
- Neo4j 负责"深挖掘"理清知识之间的关联关系

## 3. Neo4j 数据模型

### 3.1 节点类型

| 节点类型 | Label | 说明 | 示例 |
|---------|-------|------|------|
| 法规 | `:Law` | 法律法规名称 | 《噪声污染防治法》 |
| 条款 | `:Article` | 具体法条 | 第2条、第3条第1款 |
| 标准 | `:Standard` | 技术标准 | GB 3096-2008 |
| 处罚 | `:Penalty` | 处罚内容 | 罚款5000-50000元 |
| 案例 | `:Case` | 案例 | XX项目噪声超标案 |
| 主体 | `:Subject` | 当事人 | XX建筑工程公司 |
| 行为 | `:Behavior` | 违规行为 | 噪声超标排放 |

### 3.2 关系类型

| 关系 | Type | 说明 | 示例 |
|------|------|------|------|
| 包含 | `:CONTAINS` | 法规包含条款 | (噪声法)-[:CONTAINS]->(第2条) |
| 引用 | `:REFERS` | 条款引用标准 | (第2条)-[:REFERS]->(GB3096) |
| 规定 | `:DEFINES` | 条款规定行为 | (第2条)-[:DEFINES]->(噪声超标) |
| 触发 | `:TRIGGERS` | 行为触发处罚 | (噪声超标)-[:TRIGGERS]->(罚款) |
| 适用 | `:APPLIES` | 案例适用条款 | (XX案)-[:APPLIES]->(第2条) |
| 涉及 | `:INVOLVES` | 案例涉及主体 | (XX案)-[:INVOLVES]->(施工单位) |
| 施加 | `:IMPOSES` | 处罚施加主体 | (罚款)-[:IMPOSES]->(施工单位) |

### 3.3 节点属性

```cypher
// 法规节点
(:Law {
  name: String,           // 法规名称
  code: String,           // 法规编号
  source: String,         // 知识库ID
  category: String,        // law/tech/case
  created_at: DateTime
})

// 条款节点
(:Article {
  name: String,           // 条款编号
  content: String,        // 条款摘要
  law_name: String,       // 所属法规
  article_no: String      // 条款号
})

// 标准节点
(:Standard {
  code: String,           // 标准编号
  name: String,           // 标准名称
  category: String        // 声环境/空气质量等
})

// 处罚节点
(:Penalty {
  type: String,           // 罚款/责令停产等
  amount: String,         // 金额范围
  unit: String            // 元/万元
})

// 案例节点
(:Case {
  name: String,           // 案例名称
  result: String,         // 处罚结果
  case_date: Date,       // 案例日期
  source: String         // 来源
})

// 主体节点
(:Subject {
  name: String,           // 主体名称
  type: String           // 施工单位/建设单位/监管单位
})

// 行为节点
(:Behavior {
  name: String,           // 行为名称
  description: String     // 行为描述
})
```

## 4. 技术实现

### 4.1 当前实现（内存图谱）

**文件**：
- `code/backend/app/services/graph_service.py`
- `code/backend/app/api/v1/graph.py`

**核心流程**：
```
实体识别 → 向量检索 → 关系扩展 → 内存图谱构建 → LLM回答
```

### 4.2 目标实现（Neo4j 集成）

**新增文件**：
- `code/backend/app/services/neo4j_service.py` - Neo4j 操作服务
- `code/backend/app/services/graph_extractor.py` - 实体关系抽取服务

**核心流程**：
```
知识入库
   │
   ├──────────────────────────────────────┐
   ▼                                      ▼
┌─────────────────┐              ┌─────────────────┐
│  存入 MySQL/ES  │              │  异步触发抽取   │
│  (原有流程)      │              │  graph_extractor │
└─────────────────┘              └────────┬────────┘
                                           │
                    ┌──────────────────────┼──────────────────────┐
                    ▼                      ▼                      ▼
             ┌───────────┐         ┌───────────┐         ┌───────────┐
             │ 规则抽取  │         │ LLM 抽取  │         │ 关系补全  │
             │ (快速)    │         │ (语义)    │         │ (补充)    │
             └─────┬─────┘         └─────┬─────┘         └─────┬─────┘
                   └──────────────────────┼──────────────────────┘
                                           ▼
                                  ┌─────────────────┐
                                  │   写入 Neo4j    │
                                  │  (MERGE 节点+关系)│
                                  └─────────────────┘
```
