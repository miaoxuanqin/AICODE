# 知识模块-数据存储架构-技术架构设计-20260430

## 1. 概述

本文档描述住建知识库系统的数据存储架构，涵盖各存储介质的职责划分和数据流向。

## 3. 存储介质职责

### 3.1 MySQL - 元数据存储

**存储内容**：知识的基本元数据，不存储内容本身

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | int | 知识ID，主键 |
| `title` | varchar(500) | 标题 |
| `type` | enum | `pdf`/`doc`/`text` |
| `category` | enum | `law`/`tech`/`case`/`policy` |
| `source` | varchar(255) | 来源 |
| `tags` | json | 标签数组 |
| `file_path` | varchar(500) | MinIO文件路径（仅文档类型） |
| `content` | text | **不存内容** |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

### 3.2 Elasticsearch - 内容存储

**存储内容**：知识的内容文本

| 类型 | 存储内容 | 来源 |
|------|---------|------|
| `doc`/`pdf` | 文件解析后的纯文本 | PDF/Word 解析 |
| `text` | 原始文本内容 | 用户输入 |

**索引结构**：
```json
{
  "knowledge_id": "int",
  "title": "string",
  "content": "text",
  "category": "keyword",
  "tags": ["keyword"],
  "source": "string",
  "created_at": "date"
}
```

### 3.3 向量库 - 向量存储

**存储内容**：文本的向量化表示

**用途**：语义搜索、相似度匹配

**向量生成**：基于 OpenAI text-embedding-ada-002 或中文Embedding模型

### 3.4 Neo4j - 知识图谱

**存储内容**：
- 实体节点（法律、法规、标准、案例等）
- 关系边（包含、引用、属于、等）
- 实体属性

### 3.5 MinIO - 文件存储

**存储内容**：原始文档文件

**适用类型**：`pdf`、`doc`

**文件命名**：`{knowledge_id}/{timestamp}_{original_name}`

## 5. 删除流程

删除知识时需清理所有关联数据：

| 步骤 | 操作 | 存储介质 |
|------|------|---------|
| 1 | 删除元数据 | MySQL |
| 2 | 删除内容索引 | ES |
| 3 | 删除向量数据 | 向量库 |
| 4 | 删除图谱数据 | Neo4j |
| 5 | 删除原文件 | MinIO（仅 doc/pdf） |

## 6. 知识分类

| 分类 | 值 | 说明 |
|------|-----|------|
| 法律法规 | `law` | 法律、法规、条例 |
| 技术标准 | `tech` | 国家标准、行业标准 |
| 执法案例 | `case` | 行政处罚案例、事故案例 |
| 政策文件 | `policy` | 政策、通知、意见 |

## 7. 知识类型

| 类型 | 值 | 说明 | 文件存储 |
|------|-----|------|---------|
| PDF文档 | `pdf` | PDF格式 | ✅ MinIO |
| Word文档 | `doc` | Word格式 | ✅ MinIO |
| 文本 | `text` | 纯文本 | ❌ |

## 8. 索引状态

| 状态字段 | 说明 | 值 |
|---------|------|-----|
| `fullTextStatus` | ES索引状态 | `indexed`/`pending` |
| `vectorStatus` | 向量处理状态 | `done`/`pending` |
| `graphStatus` | 图谱构建状态 | `done`/`pending` |
