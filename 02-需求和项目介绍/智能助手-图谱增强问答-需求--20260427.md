> 创建时间：2026-04-27
> 需求讨论日期：2026-04-27

---

## 一、功能定位

图谱增强问答（Graph-RAG）是**基于知识图谱的多跳推理问答系统**，面向需要深度理解知识关联的业务场景。

核心价值：通过知识图谱建立实体之间的关联关系，在问答时展示推理路径，增强回答的可解释性和完整性。

---

## 二、与现有助手的关系

| 助手 | 定位 | 技术 | 特点 |
|------|------|------|------|
| 问答助手 | 通用知识问答 | RAG (ES/Qdrant) | 快速检索，单跳 |
| 执法智能助手 | 执法场景处置推荐 | RAG + 场景模板 | 领域化 |
| 工程监管助手 | 质量问题定位 | RAG + 向量搜索 | 语义理解 |
| **图谱增强问答** | **多跳推理问答** | **Graph-RAG** | **可解释推理路径** |

## 三、核心价值（解决痛点）

### 3.1 现有方案痛点

**痛点1：问答结果碎片化**

现有 RAG 返回平面知识卡片：
- 卡片1：《建设工程安全生产管理条例》第XX条
- 卡片2：脚手架安全技术规范
- 卡片3：2023年XX违规案例

用户无法直观理解知识间的关联关系。

**痛点2：无法回答多跳问题**

用户追问："这个处罚有什么依据？类似案例一般罚多少？"

现有系统只能做单跳检索，无法做多跳推理。

**痛点3：缺乏可解释性**

用户只能看到答案，无法理解答案的推导过程。

### 3.2 图谱增强解决方案

| 痛点 | 图谱增强解决方案 |
|------|------------------|
| 碎片化 | 展示知识间的关联关系（法规→条款→案例→处罚） |
| 多跳问题 | 图谱推理路径：问题→关联实体→关系推理→答案 |
| 可解释性 | 展示完整推理链，用户可追溯每个结论的依据 |

## 四、业务场景

### 场景1：执法场景多跳问答

**问**：工地脚手架坠落事故后，处罚依据和标准是什么？

**推理路径**：
```
脚手架坠落事故
    ├── 涉及法规：《建设工程安全生产管理条例》
    │       └── 包含条款：第78条（安全防护设施违规处罚）
    │               └── 导致处罚：罚款5-10万元
    ├── 类似案例：2023年XX工地事故
    │       └── 处罚金额：15万元（参照执行）
    └── 支撑标准：JGJ59-2011
```

### 场景2：施工许可前置条件查询

**问**：施工许可证申请需要哪些前置条件？

**推理路径**：
```
施工许可证
    ├── 前置条件(土地批复)
    │       └── 关联法规：《建筑法》第7条
    ├── 前置条件(规划许可)
    │       └── 关联政策：《城市规划法》
    └── 流程指引：关联办事指南
```

### 场景3：工程质量责任追溯

**问**：混凝土强度不达标是谁的责任？

**推理路径**：
```
混凝土强度不达标
    ├── 责任方：施工单位（浇筑责任）
    ├── 责任方：监理单位（监督责任）
    ├── 适用标准：GB50204-2015
    └── 类似案例：XX项目质量事故
```

## 五、图谱数据模型

### 5.1 实体类型

| 实体类型 | 代码 | 示例 | 图标 |
|---------|------|------|------|
| 问题 | question | 脚手架坠落事故 | ? |
| 法规 | law | 建设工程安全生产管理条例 | 法 |
| 条款 | article | 第78条 | 条 |
| 案例 | case | 2023年XX工地事故 | 案 |
| 政策 | policy | 建筑施工安全检查标准 | 策 |
| 处罚 | penalty | 罚款15万元 | 罚 |
| 标准 | standard | JGJ59-2011 | 标 |

### 5.2 关系类型

| 关系 | 代码 | 说明 |
|------|------|------|
| 涉及 | involves | 问题涉及的法规 |
| 包含 | contains | 法规包含的条款 |
| 导致 | leads_to | 条款导致的处罚 |
| 类似案例 | similar_case | 与问题类似的案例 |
| 处罚依据 | basis | 案例的处罚依据 |
| 支撑 | supports | 标准支撑条款 |
| 程序依据 | procedure_basis | 处罚的程序依据 |

## 七、技术方案

### 7.1 架构选择

**Phase 1: 原型验证（当前）**
- 前端：Vue3 + Element Plus + SVG
- 后端：复用现有 qa_service
- 图谱存储：MySQL（实体表+关系表）
- 图谱可视化：SVG 手绘布局

**Phase 2: 正式实现**
- 图数据库：Neo4j（可选，复杂度高时引入）
- 实体抽取：LLM API 调用
- 图谱推理：图遍历算法

### 7.2 数据流程

```
用户提问 → 实体识别 → 图谱查询 → 多跳推理 → 答案生成
                ↓           ↓           ↓
            实体边界    关系路径    推理链展示
```

### 7.3 后端接口（待开发）

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 图谱查询 | GET | /api/v1/graph/search | 根据问题查询相关实体和关系 |
| 推理路径 | GET | /api/v1/graph/reasoning | 获取推理路径 |
| 实体列表 | GET | /api/v1/graph/entities | 获取所有实体 |
| 关系列表 | GET | /api/v1/graph/relations | 获取所有关系 |
